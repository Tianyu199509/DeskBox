# Z-Order 层级管理修复

## 问题根因

### Bug #1: 点击一个格子后，其他格子跑到全屏网页后面
**根因**: `RestoreRaisedWidgetsToDesktopLayerExcept` 调用 `ForceRestoreDesktopLayerFromManager()` → `PushToBottom()` → 发送到 `HWND_BOTTOM`。有全屏浏览器时，其他格子被压到浏览器后面。

### Bug #2: 随记格子一直保持在最上方
**根因**: WidgetWindow 的 Deactivated 处理器缺少 `WidgetsRaisedFromTray` 守卫（QuickCapture 有），导致两个窗口类型行为不一致。加上鼠标钩子的抑制窗口（160ms）可能吞掉点击事件。

## 两个方案对比

| | 方案 A: 一起出现一起消失 | 方案 B: 点击的留上层，其他降级 |
|---|---|---|
| 唤起格子 | 全部出现 | 全部出现 |
| 点击其中一个 | 全部保持可见 | 被点击的留在上层，其他降为 NOTOPMOST |
| 点击外部窗口 | 全部消失 | 全部消失 |
| 复杂度 | 低（2 处改动） | 中（6 处改动） |
| 用户体验 | 简单直观，但不能单独操作一个格子 | 更自然，可以单独操作 |

### 方案 A: 全部一起出现/消失
**改动点**:
1. `WidgetManager.cs` — 鼠标钩子：点击格子时什么都不做（保持全部可见）
2. `WidgetWindow.xaml.cs` — Pointer Activation 加 `WidgetsRaisedFromTray` 守卫

**优点**: 最简单，彻底消除 Z-order 协调问题
**缺点**: 点击一个格子后，其他格子仍在上层挡住内容；用户无法单独操作一个格子

### 方案 B: 点击的留上层，其他降级（推荐）
**改动点**:
1. `Win32Helper.cs` — 新增 `SetWindowNonTopMost()` 方法（仅 `HWND_NOTOPMOST`，不 `HWND_BOTTOM`）
2. `WidgetWindow.xaml.cs` — 新增 `PushToNonTopMost()` 方法
3. `QuickCaptureWidgetWindow.xaml.cs` — 新增 `PushToNonTopMost()` 方法
4. `WidgetManager.cs:1482` — `RestoreRaisedWidgetsToDesktopLayerExcept` 改用 `PushToNonTopMost()` 替代 `ForceRestoreDesktopLayerFromManager()`
5. `WidgetWindow.xaml.cs:1248` — Deactivated 处理器加 `WidgetsRaisedFromTray` 守卫
6. `QuickCaptureWidgetWindow.xaml.cs:2393` — Pointer Activation 移除 `WidgetsRaisedFromTray` 守卫

**优点**: 符合直觉，修复根因，统一两个窗口类型的行为
**缺点**: 改动稍多

## 验证方案

1. 唤起所有格子 → 全部正常显示
2. 点击其中一个格子 → 被点击的在上层，其他仍在浏览器上方（不被压到后面）
3. 点击外部窗口 → 全部消失
4. 全屏浏览器中唤起格子 → 全部正常显示
5. 拖拽/调整大小/右键菜单期间 → 不触发层级变化
