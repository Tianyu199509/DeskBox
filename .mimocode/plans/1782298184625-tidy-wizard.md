# Fix QuickCaptureWidgetWindow Shell Migration Issues

## Problem Summary

The stage 1 refactoring moved QuickCaptureWidgetWindow from inline XAML to `WidgetShell` + `QuickCaptureWidgetContent`. The result has 4 bugs:

1. **Title bar drag broken** - PointerMoved never fires after PointerPressed
2. **Resize broken** - resize border events not working
3. **Add/More buttons non-functional** - hover shows them, but only Close click works
4. **Thick white border** - visible right-angle border around the window

## Root Cause Analysis

### Bug 1 & 2: Pointer capture not wired in host

The original XAML had `PointerPressed="TitleBarGrid_PointerPressed"` directly on the TitleBarGrid. The new code registers these via code-behind (`titleBar.PointerPressed += ...`), but:

- `TitleBarGrid_PointerPressed` (line 523) does NOT call `titleBar.CapturePointer(e.Pointer)`. Without pointer capture, the TitleBarGrid won't receive `PointerMoved` events when the cursor moves outside its bounds during drag.
- The original code also didn't capture explicitly, but it worked because the XAML-declared events were part of the visual tree event routing. Code-behind registration on a child element inside a UserControl may behave differently.

**Fix**: Add `titleBar.CapturePointer(e.Pointer)` in `TitleBarGrid_PointerPressed`, and `titleBar.ReleasePointerCapture(e.Pointer)` in `TitleBarGrid_PointerReleased`.

### Bug 3: Shell's own PointerEntered/PointerExited swallows events on TitleBarGrid

The Shell constructor (line 17-18) registers:
```csharp
PointerEntered += (_, _) => { if (ActionButtonsVisible) RightActionButtons.Opacity = 1; };
PointerExited += (_, _) => { RightActionButtons.Opacity = 0; };
```

These are on the **root Grid** of the Shell UserControl. When the pointer enters the Shell's root Grid, PointerEntered fires. But the `PointerExited` on the root Grid fires when the pointer moves from the root Grid to a child element IF the child has its own PointerEntered handler that marks the event as handled.

The host window also registers `titleBar.PointerEntered += RootGrid_PointerEntered` which sets `Shell.ActionButtonsPanel.Opacity = 1`. But the Shell's own PointerExited handler on the root Grid may fire when moving between child elements, resetting opacity to 0.

More critically: the Shell's root Grid has `Background="Transparent"` and `IsTabStop="True"`, making it a hit-test target. The ContentPresenter (Row 1) hosts the QuickCaptureWidgetContent which has its own `PointerEntered`/`PointerExited` on RootGrid. When the pointer moves from the title bar area to the content area, the Shell's root Grid PointerExited fires (pointer leaving the Grid to enter a child), resetting button opacity.

**Fix**: Remove the Shell's own PointerEntered/PointerExited handlers. The host window already manages action button visibility via its own `RootGrid_PointerEntered`/`RootGrid_PointerExited` on the TitleBarGrid.

### Bug 4: BackgroundPlate doesn't cover window edges

The Shell's `BackgroundPlate` Border has `BorderThickness="0"` and `CornerRadius="0"`, which is correct. But the Window's own default background (white/light) shows through at the edges because:

- The Shell doesn't explicitly set its own Background
- The Window's content area has a default background
- The `DwmSetWindowAttribute` with `DWMWA_BORDER_COLOR` set to `0xFFFFFFFE` (none) may not fully suppress the window frame border on all Windows versions

**Fix**: Set `Background="Transparent"` on the Shell's root Grid explicitly (already set), and ensure the Window's content background is also transparent. The real fix is likely that the `ApplyFullWindowFrame` call needs to happen after the Shell is fully loaded.

## Implementation Plan

### Step 1: Fix TitleBarGrid pointer capture in QuickCaptureWidgetWindow.xaml.cs

In `TitleBarGrid_PointerPressed` (line 523):
- After setting `_isDragging = true`, add: `((UIElement)sender).CapturePointer(e.Pointer);`

In `TitleBarGrid_PointerReleased` (line 570):
- After `_isDragging = false`, add: `((UIElement)sender)?.ReleasePointerCapture(e.Pointer);`

### Step 2: Remove Shell's own PointerEntered/PointerExited in WidgetShell.xaml.cs

Remove lines 17-18:
```csharp
PointerEntered += (_, _) => { if (ActionButtonsVisible) RightActionButtons.Opacity = 1; };
PointerExited += (_, _) => { RightActionButtons.Opacity = 0; };
```

The host window already manages button visibility via `titleBar.PointerEntered`/`titleBar.PointerExited` → `RootGrid_PointerEntered`/`RootGrid_PointerExited`.

### Step 3: Fix the white border / style issue

In `WidgetShell.xaml`, the BackgroundPlate already has `BorderThickness="0"`. The issue is likely the Window frame. Check `ConfigureWindow()` in the host:

- The `DwmSetWindowAttribute` with `DWMWA_BORDER_COLOR = 0xFFFFFFFE` should suppress the border
- `Win32Helper.ApplyFullWindowFrame(_hWnd)` is called, which extends the frame
- The Shell's root Grid has `Background="Transparent"`

Potential fix: Ensure the Shell itself doesn't introduce any default border. Check if the UserControl template adds a default Border with a background. If so, override it.

**Alternative approach**: The thick white border might be from the Window's default `Content` background. In WinUI 3, the Window's root content has a default background. Setting `this.SystemBackdrop = null` or ensuring the acrylic backdrop covers the entire window area should fix this.

### Step 4: Verify button clicks

After fixing bugs 1-3, re-test button clicks. If Add/More still don't work, check:
- Whether the Shell's Button elements are being properly created and wired
- Whether the `Click` event handler registration in code-behind is happening before the buttons are loaded
